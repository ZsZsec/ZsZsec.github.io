Vector de Ataque Identificado

Dado el contexto de un entorno de Active Directory y las credenciales de una cuenta de servicio en nuestro poder, evaluamos la viabilidad de realizar un ataque de falsificación de tickets Kerberos para impersonar a un miembro del grupo IT o directamente al usuario Administrator, lo que nos otorgaría privilegios completos sobre el servidor MSSQL

Silver Ticket Attack

Contexto del Ataque

Un Silver Ticket es un ticket TGS (Ticket Granting Service) de Kerberos falsificado que permite a un atacante impersonar cualquier usuario ante un servicio específico sin necesidad de contactar al KDC (Key Distribution Center/Domain Controller).

A diferencia de un Golden Ticket (que requiere el hash del KRBTGT), un Silver Ticket solo necesita el hash de la cuenta de servicio, lo que lo hace más sigiloso ya que:

  • No genera eventos de autenticación en el Domain Controller
  • Solo es válido para el servicio específico comprometido
  • Es más difícil de detectar por las soluciones de monitoreo tradicionales

Requisitos para el Ataque

Para generar un Silver Ticket válido, necesitamos recopilar los siguientes elementos:

  1. Hash NTLM de la cuenta de servicio: Credenciales de mssqlsvc (ya obtenidas)
  2. SID del dominio: Identificador único del dominio SIGNED.HTB
  3. SPN (Service Principal Name): Nombre del servicio al que queremos acceder
  4. Usuario a impersonar: Típicamente una cuenta con privilegios elevados (ej: Administrator)
  5. RID del usuario objetivo: Identificador relativo del usuario a impersonar

## Recopilación de Requisitos para el Silver Ticket

El primer requisito es obtener el hash NTLM de la cuenta de servicio mssqlsvc. Dado que ya disponemos de la contraseña en texto claro (purPLE9795!@), procedemos a convertirla a su representación hash NTLM:

echo -n 'purPLE9795!@' | iconv -f UTF-8 -t UTF-16LE | openssl md4

4. Estrategia de Impersonación

Para el Silver Ticket, utilizaremos una técnica que combina la impersonación del usuario Administrator (RID 500) con la inyección de membresías de grupo en el ticket PAC (Privilege Attribute Certificate).

Específicamente, incluiremos al usuario en los siguientes grupos:

  • RID 512: Domain Admins
  • RID 1105: IT (grupo con privilegios sysadmin en MSSQL)

Esta técnica es más efectiva que simplemente impersonar al grupo IT, ya que:

  • El ticket se emite para el usuario mssqlsvc (manteniendo consistencia)
  • El PAC incluye membresías a grupos privilegiados
  • MSSQL reconocerá la pertenencia al grupo IT y otorgará privilegios sysadmin

Generación del Silver Ticket

Con todos los requisitos recopilados, procedemos a forjar el Silver Ticket utilizando impacket-ticketer:

impacket-ticketer -nthash ef699384c3285c54128a3ee1ddb1a0cc -domain-sid "S-1-5-21-4088429403-1159899800-2753317549"  -domain SIGNED.HTB -spn MSSQLSvc/DC01.SIGNED.HTB -groups 512,1105 -user-id 500 mssqlsvc

Parámetros utilizados y su explicación

  • -nthash: Hash NTLM de la cuenta de servicio mssqlsvc, utilizado para firmar el ticket Kerberos generado.
  • -domain-sid: SID base del dominio SIGNED.HTB, excluyendo el RID final del usuario o grupo.
  • -domain: Nombre completo del dominio objetivo (SIGNED.HTB).
  • -spn: Service Principal Name del servicio MSSQL contra el cual se generará el ticket Kerberos.
  • -groups: RIDs de los grupos que se incluirán en el PAC del ticket.
    • 512 → Domain Admins
    • 1105 → Grupo IT
  • -user-id: RID del usuario que será representado dentro del ticket Kerberos.
    • 500 → Administrador del dominio
  • mssqlsvc: Nombre del usuario de servicio para el cual se genera el ticket Kerberos.

El ticket se guarda exitosamente en el archivo mssqlsvc.ccache.

Limitaciones del Silver Ticket Inicial

Identificación del Problema Real

Aunque logramos autenticarnos en MSSQL con privilegios sysadmin utilizando el Silver Ticket inicial, cuando intentamos acceder a recursos del sistema que requieren permisos elevados, nos encontramos con un error crítico.

El objetivo: Leer la flag de root ubicada en:

C:\Users\Administrator\Desktop\root.txt

El problema: Al intentar acceder a este archivo desde MSSQL usando OPENROWSET con el ticket generado con -user-id 500:

Análisis del Error 1346

El error 1346 de Windows indica un problema con los niveles de suplantación (impersonation) en el contexto de seguridad de Kerberos. Este error ocurre específicamente cuando:

  1. Delegación Kerberos insuficiente: El ticket no tiene los niveles de delegación necesarios para que MSSQL pueda suplantar al usuario al acceder a recursos del sistema de archivos
  2. Problema con el PAC (Privilege Attribute Certificate): El PAC del ticket no se está validando correctamente, causando que Windows rechace las credenciales al intentar acceder a recursos protegidos
  3. Discrepancia de identidad: El ticket fue generado con -user-id 500 (Administrator), pero el servicio MSSQL se ejecuta bajo la cuenta mssqlsvc (RID 1103). Esta inconsistencia entre:
    • El SID en el ticket Kerberos (Administrator - RID 500)
    • El SID real del proceso de SQL Server (mssqlsvc - RID 1103)Causa que Windows no pueda validar correctamente el contexto de seguridad y rechace la suplantación.

Generación del Silver Ticket Corregido

Solución: Alineación de Identidad + Inyección de Grupos

Para resolver el error 1346 y permitir que MSSQL acceda a recursos protegidos del sistema, necesitamos un Silver Ticket que:

  1. Use el RID correcto de mssqlsvc (1103) en lugar del RID de Administrator (500)
  2. Inyecte las membresías de grupos privilegiados en el PAC (Domain Admins e IT)
  3. Mantenga consistencia entre el SID del ticket y el SID del proceso

Esto permite que:

  • Windows valide correctamente el contexto de seguridad
  • El PAC sea reconocido como legítimo
  • Las membresías de grupo inyectadas sean efectivas
  • La suplantación (impersonation) funcione correctamente

Cambio crítico:

  • Antes: -user-id 500 (Administrator) → Causaba error 1346
  • Ahora: -user-id 1103 (mssqlsvc real) → Permite validación correcta del PAC

Parámetros explicados:

  • -user-id 1103: RID real de mssqlsvc, alineado con el proceso del servicio
  • -groups 512,1105:
    • 512 = Domain Admins (acceso total al sistema)
    • 1105 = IT (privilegios sysadmin en MSSQL)

Cómo Funciona la Solución

Flujo de autenticación corregido:

  1. El ticket Kerberos presenta el SID S-1-5-21-...-1103 (mssqlsvc)
  2. Windows valida que este SID coincide con el proceso de SQL Server
  3. Windows lee el PAC y reconoce las membresías de grupo:
    • mssqlsvc es miembro de Domain Admins (grupo 512)
    • mssqlsvc es miembro de IT (grupo 1105)
  4. Al intentar acceder a C:\Users\Administrator\Desktop\root.txt:
    • MSSQL suplanta al usuario mssqlsvc
    • Windows valida los permisos basándose en las membresías de grupo
    • Como mssqlsvc "pertenece" a Domain Admins, tiene acceso total
  5. El archivo se lee exitosamente
